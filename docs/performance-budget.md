# 성능 예산과 벤치 방법론

성능은 이 제품의 핵심 약속이다. "Obsidian/Typora보다 빠르다"를 감이 아니라 **수치**로 증명한다. 방법론은 ZeroList 벤치 원칙을 따른다: **Release 전용 측정, 실기기 없으면 "방향성만" 워터마크 강제, p50/p95/p99 보고, 고정된 비교군**.

## 무엇을 측정하는가 (성능 축)

| 축 | 정의 | 셸(Tauri)로 이기는가 | 에디터로 이기는가 |
| --- | --- | --- | --- |
| 콜드 스타트 | 실행 → 첫 편집 가능 프레임 | ✅ | — |
| 메모리(RSS) | 유휴 + 대표 문서 로드 시 | ✅ | 일부 |
| 바이너리 크기 | 배포 산출물 | ✅ | — |
| **입력 레이턴시** | 키다운 → 화면 반영 | ❌(셸 무관) | ✅ |
| 대용량 문서 편집 | 큰 문서에서 타이핑/스크롤 | ❌ | ✅ |
| 리딩 뷰 렌더 | 텍스트 → 렌더 완료 | — | ✅(Worker) |

콜드 스타트·메모리·용량은 Tauri 선택으로 이미 유리하다. **입력 레이턴시·대용량 편집**은 CM6와 라이브 프리뷰 decoration 구현의 질에 달렸다 — 여기가 진짜 승부처다([decisions.md ADR-001](decisions.md)).

## 초기 성능 예산(보수적 가드레일)

절대값은 하드웨어에 따라 다르므로 초기엔 **가드레일 + 회귀 감시**로 쓴다. 실측 후 조정한다.

```text
콜드 스타트        : 목표 < 500ms (경쟁작 Electron 대비 명확히 낮게)
유휴 메모리        : 목표 < 150MB (Electron baseline보다 낮게)
입력 레이턴시 p95  : 목표 < 16ms (한 프레임 이내), p99 < 32ms
라이브 프리뷰 켠 상태 p95 : 목표 < 16ms (decoration 재계산이 프레임을 넘기지 않게)
리딩 뷰 렌더(중간 문서) : 메인 스레드 블로킹 0ms (Worker), 결과 반영 < 100ms
대용량 문서(수만 줄) 스크롤: 드롭 프레임 없이 60fps 유지
```

## 벤치 방법론 (ZeroList 원칙)

- **Release 전용**: dev/debug 빌드는 측정하지 않는다. 최적화 빌드(프론트 프로덕션 번들 + Tauri release)만 수치로 인정한다.
- **워터마크 강제**: 실기기 벤치가 아니면(가상화·CI·개발 머신) 결과에 "방향성만(directional only)" 워터마크를 붙인다. 절대값을 신뢰하지 말고 **상대 비교와 회귀 방향**만 읽는다.
- **분포로 보고**: 단일 평균이 아니라 **p50/p95/p99**를 보고한다. 편집 레이턴시는 꼬리(p99)가 사용자 체감을 지배한다.
- **고정 비교군**: 같은 문서·같은 시나리오로 아래를 함께 측정한다.

```text
비교군:
  1. 이 에디터 (측정 대상)
  2. Obsidian   (Electron + CM6)
  3. Typora     (Electron + 자체 렌더)
  4. Logseq     (Electron)
  5. Mark Text  (Electron)  또는 순수 CM6 데모(에디터 엔진 baseline)
```

경쟁작과 **동일 문서·동일 조작**으로 재야 "이겼다"가 성립한다. 자기 자신만 재는 건 회귀 감시용이고, 경쟁 우위 주장은 비교군 측정으로만 한다.

## 입력 레이턴시 측정

```text
시나리오:
  A. 빈 문서에 연속 타이핑
  B. 중간 크기 문서(수천 줄) 중앙에서 타이핑
  C. 대용량 문서(수만 줄)에서 타이핑
  D. 라이브 프리뷰 on 상태의 A/B/C
  E. 한글 조합 입력(IME) 상태의 타이핑

각 시나리오에서 keydown -> paint 시간을 N회 수집해 p50/p95/p99 산출.
```

- 라이브 프리뷰 on/off를 **각각** 재서, decoration 재계산이 레이턴시에 얹는 비용을 분리한다.
- 한글 조합(E)은 별도로 잰다 — composition 처리 비용이 라틴 입력과 다르다.

## 벤치 하니스

```text
tools/bench/            (또는 apps/desktop/bench)
  input-latency         키 입력 → paint 수집 harness
  startup               콜드 스타트 측정
  memory                RSS 스냅샷
  report                p50/p95/p99 + 워터마크 + 비교군 표 생성
```

- `mise run bench`로 실행([development-commands.md](development-commands.md)).
- 최소 하니스는 Phase 1(macOS 앱 실행 프로토타입)에 붙인다. 초기에는 콜드 스타트, 유휴 메모리, 입력 레이턴시 smoke만 있어도 된다.
- 경쟁작 비교군 측정과 한글 조합/라이브 프리뷰 세부 시나리오는 기능이 들어오는 단계에서 확장한다.
- CI에서는 main push/수동/주기 실행으로 두고 **PR required check로는 걸지 않는다**(하드웨어 변동성 때문). 대신 회귀가 크면 리포트로 경고한다.

## 회귀 정책

- hot path(타이핑·라이브 프리뷰·대용량 스크롤)에 눈에 띄는 회귀를 넣는 PR은 근거와 완화책을 함께 보고한다([project-rules.md 성능](project-rules.md)).
- "B(캔버스)로 전환할지"의 판단도 이 벤치로만 한다: B-lite 스파이크가 CM6 대비 입력 레이턴시를 유의미하게 이기는지 실측([decisions.md ADR-002](decisions.md)).
