# 설계 의사결정 기록 (ADR)

이 문서는 왜 지금의 스택과 구조를 택했는지 남기는 단일 출처다. 코드나 스택을 바꾸려면 먼저 여기에 결정을 추가/수정하고, 기존 전략을 뒤집는 경우 사용자와 논의한다. 각 결정은 되돌릴 때 필요한 맥락을 함께 남긴다.

## ADR-001 — 셸: Tauri (vs Electron)

**결정**: Tauri 2.x를 채택한다.

**맥락**: 경쟁 대상인 Obsidian·Typora·Logseq·Mark Text는 전부 Electron + DOM 기반이다. 이들이 느리게 느껴지는 주 요인은 편집기 자체가 아니라 **Chromium을 통째로 부팅하는 콜드 스타트·메모리**다. 마크다운 에디터의 체감 성능은 축별로 갈리는데, **셸 교체로 이기는 축(콜드 스타트·메모리·바이너리 크기)과 셸과 무관한 축(타이핑 레이턴시·대용량 편집)** 이 분명히 나뉜다. 축별 표는 [performance-budget.md](performance-budget.md)를 단일 출처로 둔다.

Electron을 쓰면 경쟁작과 같은 baseline이라 "가벼움"으로 차별화가 안 된다. Tauri는 셸이 이기는 축(시작·메모리·용량)에서 즉시 체감 우위를 준다.

**트레이드오프 / 리스크**: Tauri는 OS 웹뷰를 쓰므로 플랫폼마다 렌더 엔진이 다르고 **Linux WebKitGTK가 약한 고리**다. 다만 앱 실행 프로토타입까지는 macOS 로컬 실행만 완료 기준으로 둔다([product-scope.md](product-scope.md)). Windows/Linux 검증은 프로토타입 이후 별도 단계에서 다루며, 플랫폼별 웹뷰 차이와 Linux 정책의 상세는 [tauri-shell.md](tauri-shell.md)를 단일 출처로 둔다.

## ADR-002 — 편집 표면 접근: A (vs B 캔버스, vs C 네이티브)

**결정**: **A(웹 셸 + 최고급 웹 에디터 = CodeMirror 6)** 로 시작한다.

**맥락**: 세 갈래를 검토했다.

- **A. 웹 셸 + 웹 에디터(CM6)** — 생산성 최고. 시작·메모리·용량으로 경쟁작을 이기고 편집감은 CM6로 대등 이상. 타이핑 레이턴시를 *극적으로* 이기진 못함.
- **B. 웹 셸 + 캔버스/WebGL 자체 렌더 에디터** — 렌더 파이프라인 직접 통제. 단 캔버스는 IME 직접 접근 불가(숨은 input + composition 해킹), 접근성 트리 부재, 가변폭 텍스트 셰이핑을 직접 구현. B-full로 갈수록 비용은 C에 수렴하는데 천장은 C보다 낮다.
- **C. 완전 네이티브(Zed류, GPU 렌더 + rope)** — 성능 천장 최대, 비용 최대. 한글 IME·레이아웃·접근성 전면 재구현. 사실상 다년 프로젝트.

**한글 IME**가 결정적 제약이었다. B/C는 조합형 입력을 직접 배선해야 하고(캔버스/네이티브 모두 악명 높음), A(CM6/contenteditable)는 브라우저가 조합을 처리해 준다.

**되돌림 경로**: 에디터를 별도 패키지(`packages/editor`)로 격리하고 앱은 인터페이스에만 의존한다. 나중에 B가 필요해지면 그 패키지만 교체한다. 단, "A 완성 후 B 이전"은 매몰비용 함정이 있으므로, B로 전환할 근거는 **성능 실측(p50/p95/p99)** 으로만 판단한다([performance-budget.md](performance-budget.md)).

## ADR-003 — 진실의 원천: 텍스트 (vs 문서 트리)

**결정**: **마크다운 텍스트 문자열을 진실의 원천**으로 둔다(제품 정체성 = "마크다운을 다루는 도구").

**맥락**: 두 패러다임은 원천이 다르다.

- **텍스트 원천(Obsidian식 라이브 프리뷰)**: 원천 = 마크다운 문자열. 렌더는 위에 얹는 decoration. 원본 서식 100% 보존.
- **트리 원천(Typora식 WYSIWYG)**: 원천 = 구조화 트리. 마크다운은 (역)직렬화 포맷 → **왕복이 손실적**(`*`↔`_`, 공백, 리스트 마커 정규화). 파워유저가 싫어함.

사용자는 "원본 마크다운 서식 보존이 중요한, 파워유저용 도구"를 택했다 → 텍스트 원천.

**따라오는 이점**: 하나의 텍스트 원천 위에 **소스 모드 ⟷ 라이브 프리뷰 ⟷ 리딩 뷰**를 거의 공짜로 얹을 수 있다(전부 CM6 + 렌더 레이어, 원천 하나, 왕복 손실 없음). 즉 "라이브 프리뷰냐 WYSIWYG냐"의 실용적 대부분을 이 하나의 선택으로 얻는다.

**보류**: 진짜 Typora식(마크다운 안 보이는 워드프로세서, ProseMirror 트리 원천)은 두 번째 엔진 + 손실적 브릿지라 초기 범위에서 제외한다. 강한 수요가 확인되면 별도 모드로 재검토.

## ADR-004 — 파서: unified(remark/rehype) + 인에디터 Lezer

**결정**: 리딩 뷰·익스포트는 **unified(remark → rehype)**, 편집 화면 실시간 하이라이트는 **CM6 내장 Lezer**로 이원화한다.

**맥락**: 편집 화면은 매 키 입력마다 재파싱하므로 **증분 파서**가 필수다(Lezer가 이를 제공, CM6에 통합됨). 리딩 뷰/익스포트는 전체 렌더가 필요하고 플러그인 생태계(GFM·수식·mermaid·sanitize)가 풍부한 unified가 적합하다. 단 전체 렌더는 무거우므로 **Web Worker에서 실행**해 타이핑을 막지 않는다([parser-pipeline.md](parser-pipeline.md)).

**보안**: HTML 렌더 경로는 `rehype-sanitize`를 기본 on으로 둔다(신뢰할 수 없는 문서의 XSS 차단).

## ADR-005 — 모노레포: Bun 워크스페이스 + Cargo 워크스페이스

**결정**: JS/TS는 Bun 워크스페이스, Rust는 Cargo 워크스페이스로 한 리포에서 관리한다.

**맥락**: 에디터 엔진·마크다운 파이프라인·UI 킷을 앱과 분리된 패키지로 두면 경계가 명확해지고(ADR-002의 되돌림 경로), 재사용·테스트가 쉬워진다. 구조는 [project-structure.md](project-structure.md)를 단일 출처로 둔다.

**결정**: FSD를 채택하고 레이어 간 import 방향을 `steiger`(옵션)로 강제한다.

**맥락**: 에디터는 기능(파일 열기·저장·검색·프리뷰 토글·설정)이 빠르게 늘어난다. FSD의 레이어(app→pages→widgets→features→entities→shared) 단방향 의존 규칙이 스파게티를 막는다. 상세는 [frontend-fsd.md](frontend-fsd.md).
